## Draft Spec

Intent (preserved): rebuild the `sr-math-lesson` skill to be **JSONB-first** so the DB JSONB is the single content authority. The generator must (a) read the per-stage concept ledger from `sr_content_ledger`, not from `resources/content/math-ledger/*.json`; (b) author the lesson **content** (card-tree JSONB) and **exercises** (deck JSONB) and persist them to `sr_lessons.content` / `sr_lessons.exercises`; (c) render HTML + PDF **from that JSONB** as derived caches; (d) author each substantial card's **read-check** items into the card's `read_check[]`. Answer KEY stays neutral-base-only and never reaches learner-visible HTML.

Scope (this ticket, generator only):
- The `.agents/skills/sr-math-lesson/` skill (SKILL.md, references, scripts, template) is rebuilt to the JSONB-first pipeline.
- Ledger authoring persists a ledger document to `sr_content_ledger (subject, stage, ledger, src_rev)`.
- Content authoring produces `sr_lessons.content = { cards: [ { id, num, anchor, rev, body[], read_check[] } ] }` honoring the documented shape (stable node ids; learner-visible `num`; genre anchors 概念课 motivation/model/anatomy/boundary/connections/oral, 方法课 motivation/explain/examples/connections/oral, 练习课 motivation).
- Exercise authoring produces `sr_lessons.exercises = { items: [ { id, ord, type, mode, layer, review_of, key, rev } ] }` honoring the documented shape (`mode` ∈ choice|input|work; `key` ∈ {correct_index}|{accept}|{answer}; `review_of` for 复习).
- Learner-visible **prose** (card body text, read-check prompts/options, exercise prompts/options) is written to the source-locale overlay `sr_lesson_i18n (lesson_id, locale='zh', overlay)`, keyed by node_id, per the documented neutral-base + overlay split (zh is itself the SOURCE overlay row). Formulas (KaTeX), SVG, numeric literals, and the answer KEY stay in the neutral base and are never duplicated into the overlay. [This is the pivotal architecture point — see Grill Required + Risks.]
- A deterministic **render** capability turns (content + zh overlay + lesson metadata) into the self-contained lesson HTML (reusing `assets/lesson-template.html`), and a print PDF (best-effort playwright-core), stored as the derived caches `sr_lessons.html` / `sr_lessons.pdf`.
- Preserve the existing pedagogy contracts: ledger prerequisite-closure check, genre section structure/anchors, the produce → independent-review → deterministic-save three-phase discipline.
- Answer-key secrecy: `correct_index`/`accept`/`answer` live only in the neutral base JSONB `key`; the rendered HTML (and any learner-visible projection) contains no KEY.

Non-scope (explicitly out — do NOT build here):
- Migration of the 16 existing lessons + the 2 on-disk ledgers into JSONB (STEMROBIN-21).
- The card-by-card reading UI / soft-gate flow (STEMROBIN-22).
- `en` translation overlay / translation skill (STEMROBIN-23) — only the `zh` SOURCE overlay is authored here.
- Any `app/` code change, any `Dockerfile` change, any new dependency, any new recurring cost.
- Deleting/altering the `sr_users` credential row or the 16 real lessons' existing data.

Compatibility requirements:
- The rebuilt saver keeps writing a valid `sr_lessons.html` (derived) so the existing app reader/PDF path keeps working unchanged.
- `sr_lessons.content` / `.exercises` shapes match the STEMROBIN-19 documented contract exactly (Postgres does not enforce JSONB internals; the generator/saver is the enforcement point).
- Existing `sr_questions`-based lessons are untouched by this ticket (this ticket does not migrate them; the sample uses a disposable id/stage).

Acceptance (from intent):
1. Using the rebuilt generator, author ONE small representative sample math lesson → DB has its ledger (`sr_content_ledger`), card-tree `content` and exercise `exercises` JSONB on an `sr_lessons` row; render readable HTML + a PDF FROM that JSONB.
2. Every substantial card in the sample has a `num` and ≥1 read-check; the rendered HTML contains NO answer KEY.
3. The generator completes the above WITHOUT reading any local ledger file.

## Code And Evodocs Findings

- **Current pipeline is HTML-first, file-ledger, `sr_questions`-deck** (`save-lesson.mjs`): authors author full lesson HTML directly; the deck is a flat JSON array persisted to `sr_questions`; a `practice` `<section>` is generated FROM the deck and injected into the stored HTML (prompts + options only, KEY excluded — lines 145–182, 258–264); PDF is rendered via playwright-core best-effort (lines 190–210). The rebuild inverts this: JSONB is authored first; HTML/PDF/practice become render outputs of the JSONB. The existing KEY-free practice projection (`renderPractice`, the `esc()` escaping, `optionLabel()`) is directly reusable as the render-from-JSONB projection.
- **Answer-key secrecy is already a load-bearing invariant** in three places: `save-lesson.mjs` never writes `answer`/`correct_index`/`accept` into `renderPractice` output; `check-exercises.mjs` forbids `accept` on choice items; charter `engineering-rules.md` + `mod--database-schema.md` "Answer-key secrecy is a schema-plus-service invariant". The JSONB contract continues this: `key` is neutral-base-only, overlay "MUST NEVER contain an answer KEY". The rebuild must keep the KEY out of both the overlay and the rendered HTML.
- **Genre anchors are a fixed, validated contract.** `save-lesson.mjs:89` `ANCHORS` (概念课 6 / 方法课 5 / 练习课 1) and `lesson-contract.md` define the ordered section set per genre; `.tmp/plan-card-reading.md` §1.2 confirms these anchors are the "natural card boundaries". The card-tree `content.cards[].anchor` reuses exactly this anchor set; `num` maps to the existing `sr-sec-num` learner-visible numbering.
- **The STEMROBIN-19 JSONB contract (verified live).** psql read-only probe (schema `stemrobin-schema`): `sr_content_ledger`, `sr_lesson_i18n`, `sr_content_answer_events` tables and `sr_lessons.content` / `sr_lessons.exercises` columns all EXIST. `sr_content_ledger` currently has 0 rows and 0/16 lessons have `content` — i.e. STEMROBIN-19 was schema-only and migration (STEMROBIN-21) has not run. Consequence for this ticket: "read ledger from DB" presupposes a ledger row exists, so the rebuilt generator must be able to WRITE a ledger to `sr_content_ledger` (ledger authoring → DB), and the sample must first save its sample ledger before authoring content. This matches G1/G7 ("三类 JSONB 全部入库"; "ledger 入库").
- **The documented `content.body` prose contract is the pivotal finding.** `stemrobin.sql:226-227` states card `body[]` prose nodes are "node-id refs (text in overlay)"; formulas/SVG are neutral and inline. `stemrobin.sql:259-265` states zh is the SOURCE locale and "is itself an overlay row", and the overlay holds ONLY prose. So rendering readable HTML "from the JSONB" requires joining neutral `content` + the `zh` overlay. The ticket's own write-list phrases it as "content/exercises"; reconciling that with the schema's prose-in-overlay contract is the one architecture decision worth surfacing (see Grill Required). Basis for the recommended resolution: the schema is the SSOT contract, §11 干净设计原则 forbids "leftover compromise", engineering-rule #5 (SSOT/one-way), and STEMROBIN-23 can only ADD an `en` overlay if the `zh` source overlay already exists — all point to authoring the `zh` overlay now and joining at render.
- **R-UI (peer/best-practice + touched surfaces).** No `app/` UI surface changes in this ticket. The only rendered surface is the derived lesson HTML, which must remain byte-compatible-in-spirit with today's `sr_lessons.html` (same `lesson-template.html` head, DESIGN teal/green/white tokens, `data-sr-section`/`sr-sec-num`/`sr-sec-name` structure, KaTeX CDN wiring) so the existing sandboxed-iframe reader (`mod--app--learner-experience`) and PDF download keep working with no reader change. Industry pattern this mirrors: **headless/structured-content + derived presentation** (CMS "content as data, render to HTML/PDF/other channels") — the same pattern the plan calls "HTML/卡片/PDF/各语言全部从 JSONB 派生". The adjacent surface that becomes stale is the on-disk `resources/content/math-ledger/*.json` (no longer authoritative once ledger reads move to DB) — but migrating the existing on-disk ledgers is STEMROBIN-21, so within THIS ticket the on-disk files simply stop being the generator's read source; they are not deleted here.
- **R-EXT (no new external interface).** No new external API/SDK/cloud/auth provider. The "new interface" is the in-repo JSONB contract (documented + live-verified above). `postgres` (server-only, `.env` `EASYAPP_DATABASE_URL`, `search_path="stemrobin-schema"`, ssl require) and `playwright-core` (best-effort PDF) are established repo deps; the operation cross-check (upsert semantics on `sr_content_ledger` PK `(subject,stage)`, `sr_lesson_i18n` PK `(lesson_id,locale)`, `sr_lessons` id conflict) is satisfied by ordinary `ON CONFLICT` upserts the existing saver already uses.
- **`.agents/skills/node_modules` is NOT installed in this worktree** (only `package.json` present). cap6 must run `cd .agents/skills && npm install` (established one-time setup per runbook) before running any saver/render script. No new dependency is added.

Basis: plan D1 ("只给实质卡：motivation/model/anatomy/boundary…；跳过 oral / 练习课"). Low-risk default; exact set finalized in cap5.
- A2: The sample uses a disposable id/stage that maps to NO human course-guide entry, so the human-outline fidelity check (`check-outline.mjs`) does not apply to the sample; the outline check remains enforced for real stages. Basis: task says use a clearly-disposable sample id; outline fidelity is about the real curriculum guide. Low-risk (implementation latitude).
- A3: Exercises deck stays choice-first for the sample (current SKILL commitment #4 + `backfill-choice-decks.mjs` evidence of the choice-only migration), while the shape validator accepts the schema's full `mode` set (choice|input|work) so the JSONB contract is honored without weakening pedagogy. Low-risk.
- A4: The reference answer/explanation (`key.answer`, and any read-check post-answer explanation) is KEY-class (post-answer reveal) and therefore stays in the neutral base `key`, NOT the overlay — consistent with the schema (`overlay` MUST NEVER contain `answer`). - A5: `sr_content_answer_events` is not written by the generator (it is a learner-runtime table); the generator only writes ledger/content/exercises/zh-overlay + derived html/pdf. Low-risk.

## Risks

- **DB/schema (pivotal): prose-in-overlay vs render-from-content.** If the render path is built to read only `content`/`exercises` with prose inlined, it contradicts the documented `content.body` = node-id-refs contract and creates a leftover compromise that STEMROBIN-21/23 must undo. Mitigation: author the `zh` source overlay and join at render (recommended). This is the one item that could materially change files touched + the data contract → routed to grill.
- **DB (prod-data safety, redline #2).** Writing to the shared Azure Postgres. Must use a disposable sample id/stage, must DELETE sample rows after proof, and must never touch `sr_users` or the 16 real lessons' data. Deck/content saves must be scoped to the sample id only.
- **Answer-key leakage.** The KEY must be excluded from BOTH the `zh` overlay AND the rendered HTML/practice projection. Regression risk if the render projection accidentally serializes `key`. Mitigation: reuse the existing KEY-free `renderPractice` pattern; add a grep-for-KEY empirical check in cap6.
- **Scope creep.** The JSONB-first rebuild is a large surface (ledger saver, content author+validate, exercises author+validate, overlay, render, save). Must stay within the generator; must NOT drift into migration (STEMROBIN-21), reading UI (STEMROBIN-22), or translation (STEMROBIN-23). Mitigation: only the `zh` source overlay is authored; no `app/` change; one disposable sample.
- **Contract enforcement is generator-side only.** Postgres does not constrain JSONB internals (stemrobin.sql:213-215). A shape bug persists silently. Mitigation: deterministic `check-content.mjs` + adapted `check-exercises.mjs` gate before any DB write; empirical psql readback in cap6.
- **R-TEST dev-time obstacles (named):**
  - `.agents/skills/node_modules` not installed → must `cd .agents/skills && npm install` before any run (no new dep, just setup).
  - Shared Azure Postgres must be reachable with valid `.env` `EASYAPP_DATABASE_URL` (verified reachable now via psql; if it becomes unreachable in cap6 → blocked exit, record in `## Blocker`, no mocks).
  - PDF via `playwright-core` is best-effort; a browser-launch failure can leave `pdf` empty. Acceptance needs a real PDF, so cap6 must confirm the browser engine launches (Chrome/Chromium) or record the blocker; do not fabricate a PDF.
  - Skills have no vitest/test runner → deterministic-script unit coverage + the empirical sample run + psql readback + KEY grep are the verification surface (no automated CI harness for skills).
  - Sample cleanup must be reliable (delete sample `sr_lessons` / `sr_content_ledger` / `sr_lesson_i18n` rows) so the proof does not pollute the catalog.
