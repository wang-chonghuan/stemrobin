## Implementation Approach

Schema-only, additive DDL appended to the single SSOT file `ssot-schemas/db-schemas/stemrobin.sql`, reusing its existing style (`SET search_path TO "stemrobin-schema"` header already present; `CREATE TABLE/INDEX IF NOT EXISTS`; `ALTER TABLE ADD COLUMN IF NOT EXISTS` as already used for `sr_answer_events.attempt_id`). Concrete structures (satisfying R1–R6):

1. **Neutral base content (R2, R3) — extend `sr_lessons`** (reuse existing lesson identity row; per grill recommended default):
   - `ALTER TABLE sr_lessons ADD COLUMN IF NOT EXISTS content JSONB;` — the neutral card tree.
   - `ALTER TABLE sr_lessons ADD COLUMN IF NOT EXISTS exercises JSONB;` — the neutral deck.
   - SQL comments must document the JSONB contract: `content` = `{ cards: [ { id, num, anchor, rev, body: [<neutral nodes: formula/svg/prose-ref by node_id>], read_check: [ { id, mode:'choice'|'input', key: { correct_index? , accept?[] }, rev } ] } ] }`; `exercises` = `{ items: [ { id, ord, type, mode:'choice'|'input'|'work', layer, review_of?, key: { correct_index?|accept?[]|answer? }, rev } ] }`. Comments must state: prose text is NOT here (it lives in the overlay, keyed by node_id); KEY lives ONLY here; `html`/`pdf` on this table are derived caches, SSOT = these JSONB columns. Note: existing rows keep NULL until a later migration ticket.
2. **Per-locale overlay (R4, R5) — new table**:
   - `CREATE TABLE IF NOT EXISTS sr_lesson_i18n ( lesson_id TEXT NOT NULL REFERENCES sr_lessons(id) ON DELETE CASCADE, locale TEXT NOT NULL, overlay JSONB NOT NULL, updated_at TIMESTAMPTZ NOT NULL DEFAULT now(), PRIMARY KEY (lesson_id, locale) );`
   - Comment: `overlay = { "<node_id>": { "t": "<translated prose>", "src_rev": <int> } }`; prose only, NEVER any KEY field (`correct_index`/`accept`/`answer`); `zh` is the source locale represented here; staleness ⇔ `overlay[node].src_rev < base node.rev` or node absent.
3. **Per-stage ledger (R1) — new table**:
   - `CREATE TABLE IF NOT EXISTS sr_content_ledger ( subject TEXT NOT NULL, stage INT NOT NULL, ledger JSONB NOT NULL, src_rev BIGINT NOT NULL DEFAULT 1, updated_at TIMESTAMPTZ NOT NULL DEFAULT now(), PRIMARY KEY (subject, stage) );`
   - Comment: stores the `resources/content/math-ledger/stage-*.json` document verbatim; authoring metadata (source `zh`), not part of the learner i18n overlay.
4. **Card/exercise answer events (R6) — new table**:
   - `CREATE TABLE IF NOT EXISTS sr_content_answer_events ( id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, user_id BIGINT NOT NULL REFERENCES sr_users(user_id) ON DELETE CASCADE, lesson_id TEXT NOT NULL REFERENCES sr_lessons(id) ON DELETE CASCADE, kind TEXT NOT NULL CHECK (kind IN ('read_check','exercise')), node_id TEXT NOT NULL, is_correct BOOLEAN, chosen INT, answer_text TEXT, locale TEXT, created_at TIMESTAMPTZ NOT NULL DEFAULT now() );`
   - `CREATE INDEX IF NOT EXISTS sr_content_answer_events_user_lesson_idx ON sr_content_answer_events (user_id, lesson_id);`
   - Comment: `node_id` is the JSONB item identity (read-check item id or exercise item id) — no FK to a relational question row; disposable data (seed D12); one table, `kind` discriminates read-check vs exercise.

The `REFERENCES sr_users(user_id)` in step 4 is a foreign-key declaration on a NEW table; it reads the users table's key but does not alter/delete any `sr_users` row — R8 preserved.

## Implementation Drift Controls

- **Additive-only, no destruction (R7, redline #2):** cap6 must add only `CREATE ... IF NOT EXISTS` / `ALTER TABLE ADD COLUMN IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS`. No `DROP`, no `ALTER COLUMN ... TYPE`, no `DELETE`/`UPDATE` against real rows, no change to any existing table's existing columns. If any planned change cannot be expressed additively, stop and record it in `im-handoff.md` `## Blocker` — do not force it.
- **`sr_lessons` new columns via `ALTER` (R7, Critical Existing Contract):** because `sr_lessons` already exists in prod, cap6 MUST add `content`/`exercises` with `ALTER TABLE ADD COLUMN IF NOT EXISTS`. Editing the `CREATE TABLE IF NOT EXISTS sr_lessons` block instead is a defect (it would no-op on the existing table and the columns would never appear) — verify by querying `information_schema.columns` after apply.
- **KEY isolation is structural (R3, R4, answer-key-secrecy contract):** the overlay table/JSONB must never carry `correct_index`/`accept`/`answer`. Cap6 must prove absence by querying the demo overlay rows for those keys and expecting none. Do not add any KEY column to `sr_lesson_i18n`.
- **`sr_users` untouched (R8):** the only reference to `sr_users` is the new event table's FK. Cap6 must verify the row count/emails/hashes are identical before and after apply. Never `DELETE`/`UPDATE`/`ALTER` `sr_users`.
- **No app/skill/story changes (Non-Scope):** cap6 edits only `ssot-schemas/db-schemas/stemrobin.sql` (plus ticket-scoped test artifacts under the ticket `tests/`). No `app/src/lib/*`, no `.agents/skills/*`, no story tables.
- **Rejected option stays absent (BD1):** do not add any prose column to the neutral base and do not store `zh` prose inline; `zh` prose belongs in `sr_lesson_i18n`.
- **Demo rows are disposable and isolated (R9):** verification uses a clearly-namespaced throwaway lesson id (e.g. `zzz-schema-verify-19`) so it cannot collide with real content; clean it up (DELETE the demo rows it created) after capturing outputs, or leave it only if clearly namespaced — but never touch real content or `sr_users`.
- **Staleness convention (R5):** cap6 documents `rev`/`src_rev` in SQL comments and demonstrates it in the demo rows; it must NOT try to DDL-enforce JSONB internals.

## Phases

1. **Append DDL.** Add the four structures (steps 1–4 above) to `ssot-schemas/db-schemas/stemrobin.sql` with the documented comments. Verify: the file still has the `SET search_path` header once at top; new statements are all additive/idempotent. No existing statement is modified.
2. **Apply to shared DB.** From the worktree root: `set -a && . ./.env && set +a && psql "$EASYAPP_DATABASE_URL" -f ssot-schemas/db-schemas/stemrobin.sql`. Verify: command exits 0; re-run once to confirm idempotency (still exits 0, no errors). Regression check for the shared store: this only creates tables/columns/indexes and touches no existing rows.
3. **Structure existence checks (R1–R6).** psql `information_schema` queries prove: `sr_lessons.content` and `sr_lessons.exercises` columns exist; `sr_lesson_i18n`, `sr_content_ledger`, `sr_content_answer_events` tables exist with expected columns and the `kind` CHECK.
4. **Seed disposable demo + AC queries (R9).** Insert one namespaced demo lesson row with neutral `content` (a card with a read-check `key`) + `exercises` (a choice item with `key.correct_index`, an input item with `key.accept`, a work item with `key.answer`); one `sr_content_ledger` row; `zh` + `en` `sr_lesson_i18n` overlays holding prose only; one `read_check` and one `exercise` `sr_content_answer_events` row. Then run the AC queries: (a) per-stage ledger; per-lesson card-tree + deck; per-locale overlay; answer events — all return the demo data. (b) KEY isolation: query the neutral base for the keys (present) and query every overlay row's JSONB for `correct_index`/`accept`/`answer` (must return zero). Regression check: FKs resolve, `ON DELETE CASCADE` demo cleanup removes only demo rows.
5. **`sr_users` integrity (R8).** Query `SELECT count(*), array_agg(email ORDER BY user_id), array_agg(left(password_hash,10)) FROM sr_users;` before (from cap3 baseline: 2 rows) and after; assert identical. Confirm the demo insert/cleanup never referenced `sr_users` except via the new FK.
6. **App regression guard.** `cd app && npm run test` (vitest) — must pass unchanged (app code untouched). This confirms the additive DDL breaks no existing contract the unit suite covers.
7. **Cleanup + handoff.** Delete the disposable demo rows (they cascade cleanly). Write `im-handoff.md` with the captured psql outputs (structure checks, AC queries, KEY-isolation proof, `sr_users` before/after), the vitest result, actual file changes, spec/plan alignment, and any residual notes for downstream tickets.

## Unit Test Plan

The empirical psql verification (phases 2–6) is the primary, highest-risk coverage and its outputs are the acceptance evidence — captured verbatim in `im-handoff.md`. This is a schema-only ticket applied to the shared live DB (there is no local disposable Postgres and no `nf-db` skill; the runbook's psql path is the DB access method), so the "unit" surface is the DDL + queries, not app code.

Ticket-scoped test artifacts live under the ticket `tests/` directory. Provide a re-runnable verification script (e.g. `tests/verify-schema.sh` or `.mjs`) that: applies the DDL, runs the structure-existence checks, seeds + queries the disposable demo, asserts KEY isolation (zero KEY fields in any overlay), asserts `sr_users` unchanged, and cleans up — so the AC can be reproduced. High-risk assertions it MUST include (not happy-path only):

- Structure existence: all four structures present after apply (R1–R6); `sr_lessons.content`/`exercises` added as columns (not silently skipped) — the `ALTER ADD COLUMN` correctness check (R7).
- KEY isolation (R3, R4): neutral base contains the KEY; every overlay (`zh` and `en`) contains none of `correct_index`/`accept`/`answer`.
- Answer-event queryability + discriminator (R6): both `read_check` and `exercise` rows insert and query; the `kind` CHECK rejects an invalid value.
- Idempotency (R7): applying the DDL twice both succeed with no error.
- `sr_users` integrity (R8): row count/emails/hashes identical before and after; login-bearing row untouched.
- No-regression: `cd app && npm run test` passes (shared app consumers of the untouched tables).
- Rejected option absent (BD1): no prose column on the neutral base; `zh` prose only via overlay.

If a check cannot run (DB unreachable / credentials invalid), that is a blocked external premise: stop, record evidence in `im-handoff.md` `## Blocker`, do not mock around it.
