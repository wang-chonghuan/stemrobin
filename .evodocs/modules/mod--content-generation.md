# purpose

The content-generation module is StemRobin's controlled authoring pipeline for
the learning material that the application serves from PostgreSQL. It converts
human course intent and public-domain source material into persisted lessons,
practice decks, biography chapters, questions, and printable PDFs. The module is
not a runtime content directory: the web application does not read these skill
instructions or source artifacts to show a learner a lesson. It reads the rows
written by the skills after their authoring and validation process succeeds.

The pipeline is designed to keep two kinds of work separate. A model or agent
authors meaning-rich educational content, while a different reviewer evaluates
whether it meets the learner-facing contract. Deterministic scripts then check
structural facts and carry out the database mutation. This producer, gate, and
saver separation matters because lesson quality, prerequisite closure, source
provenance, and answer-key secrecy are not reliably preserved by a single
ungated generation pass.

The module has two active child responsibilities. Math courseware produces
concept-led lessons and recall-oriented practice. Biography reading produces
public-domain narrative chapters and comprehension or reflection questions. They
share operational ownership of draft content, database-only persistence, PDF
generation, root environment secrets, and answer data that must remain hidden
until a learner responds. They intentionally differ in their evidence and
pedagogical contracts.

# structure

The parent layer consists of the active skill family, its dedicated dependency
package, shared human guidance, and one legacy math generator. The dependency
package provides `postgres`, `marked`, and `playwright-core` for the persistence
scripts. It is installed separately from the web application dependencies so
content generation can run from the repository root without coupling authoring
tools to the container image. Scripts locate the repository root through Git,
read the ignored root environment file, and use the project schema in the shared
database. They never echo the connection secret.

Human-authored content references define the starting point. The common course
guide requires standard terms and rigor with smaller conceptual steps, rejects
fluff and premature concepts, and treats recognition, representation, operation,
reverse reasoning, and error analysis as necessary learning evidence. The math
course guide provides ordered stages and titles. The design reference supplies
the visual vocabulary used by rendered courseware. These documents constrain
generation, but they are not mutable output generated by the skills during a
content run.

The math child is structured around a human outline followed by a
machine-readable concept ledger. The ledger records a stage model, prior assumed
knowledge, each lesson's introduced and consumed terms, genre, boundary cases,
and review targets. From it, separate authoring steps produce lesson HTML and
structured exercise data. Independent gates evaluate ledger pedagogy, lesson
clarity, and deck quality. Deterministic checks validate outline fidelity,
prerequisite closure, exercise composition, metadata, and persistence shape.
The saver writes the lesson, deck, and printable representation.

The biography child is structured around provenance and narrative. It converts a
public-domain book to working Markdown, adapts it into a chapter outline, writes
one long Chinese narrative chapter at a time, generates questions, gates the
chapter, and saves it. The narrative is not a disguised lesson: it follows events
in prose, leaves reflection and ethical reasoning to questions, and records the
source URL on the story. The saver checks chapter form and section sequence,
upserts the story and chapter, replaces chapter questions, and renders a PDF.

The older `sr-lesson` skill remains as parent-owned historical context. Its own
description says `sr-math-lesson` replaces it. Some of its instructions still
refer to static HTML paths, a five-section model, and a narrower choice/work
deck, even though the active math pipeline uses ledgers, input mode, review
layers, generated practice injection, and current database columns. Its script
can write current lesson rows but it is not the coherent source for new math
content.

# flows

Every active content flow starts by choosing the correct skill and reading its
shared contract. For math, the author first anchors work in the human outline.
If the stage needs a new or revised concept ledger, the ledger declares the
central mental model and the exact dependency graph for its lessons. The outline
checker prevents silently replacing or reordering human lesson titles. The
ledger checker rejects any lesson that consumes a term not previously introduced
or explicitly declared as prior knowledge. This turns hidden prerequisite debt
into a visible authoring decision.

Once a ledger is valid, the math author creates a single lesson in the genre
specified by that ledger. Concept lessons name and parse the parts of a model,
teach positive and negative examples, and cover boundary cases. Method lessons
derive an operation from its principle and require worked reasoning. Practice
lessons introduce no content and orient the learner to retrieval work. A separate
deck author creates sixteen to twenty-four items with recognition, operation,
error analysis, explanation, and review responsibilities. Typed input is
preferred for short answers; choice is reserved for diagnostic distinctions; work
captures explanation. A deck's review items target earlier ledger terms, making
spacing a property of generated content rather than a runtime scheduler.

After independent gates pass, the math saver validates the id against the human
outline and ledger, validates the lesson's appropriate section anchors, and
upserts the lesson HTML, metadata, and a best-effort PDF. The deck save validates
question shape and composition, replaces all question rows for that lesson, and
regenerates a prompt-only practice section inside stored lesson HTML. It then
re-renders the PDF. Re-saving lesson HTML later removes that injected section,
so a deck save must follow it to restore the learner-visible practice and final
print output.

Biography work begins with a source that is out of US copyright. The source is
converted into working Markdown by a deterministic wrapper around `markitdown`;
the author produces an outline that fits the target reader rather than copying a
table of contents. A chapter author writes continuous Chinese narrative prose
from real source detail, avoids quotes, HTML, and lists, and meets the minimum
Chinese-character length. Questions are mostly detail-sensitive choices with at
least two open reflection or judgment prompts. The gate rejects a dull lesson
format, unsupported facts, padding, or sermon-like prose.

The biography saver checks identifiers, question modes, narrative structure,
length, absence of quotes and lists, and consecutive numbered sections. It reads
previous saved chapters for the same story and rejects a chapter whose first
section does not continue the prior global number. On success it upserts story
provenance and chapter text, replaces questions, and stores a printable PDF.
Repeated saves are intentional overwrites for the same content identifier.

# module-relationships

The database-schema module is the persistence contract for every final output.
Math writes `sr_lessons` and `sr_questions`; learner attempts later attach to
their question ids. Biography writes `sr_stories`, `sr_story_chapters`, and
`sr_story_questions`, with separate answer-event tables. Both pipelines rely on
the schema's answer-mode and foreign-key contracts. Schema changes therefore
require coordinated updates to validators, savers, and the application's
server-side readers before new content is persisted.

The application domain-services child is the downstream runtime consumer. It
uses lesson ids to activate the fixed curriculum outline, fetches stored lesson
HTML/PDF and story Markdown/PDF, sends only public question fields to the
browser, and performs server-side answer judgment. The learner-experience child
then renders the resulting courseware in an iframe or reading surface. Content
generation must never compensate for a runtime behavior by hand-editing database
rows or making answers visible in rendered HTML.

The human guides are upstream constraints. The math ledger is a downstream
machine source for terms and review order, while the course guide remains the
authority for human title order and instructional direction. The story source
URL is persisted so provenance survives outside a transient generation session.
The shared design reference constrains classroom-facing visual language but does
not turn skills into UI code.

The active child skills have isolated detailed responsibilities. Math courseware
owns ledger closure, instructional genres, deck composition, and practice
injection. Biography reading owns public-domain verification, narrative form,
chapter section continuity, and story-specific question semantics. The legacy
skill stays inside this parent because it is superseded and should not create a
parallel current workflow.

# constraints

Final content persistence is database-only and occurs only through the
deterministic saver scripts. Do not hand-write `sr_*` rows, edit a saved course
artifact in place, or create a second publishing path. Persist only after the
specified independent gate has passed. New content defaults to `draft`; the
current application reads those fields without status filtering, so the
promotion policy must be treated as an operational decision rather than assumed
to be enforced by the reader.

The root environment file and database secret are server-side operational
inputs. The skills must retain their separate dependency package and must not
become part of the application Docker runtime. PDF generation is best effort
when a browser engine is unavailable, but the saver must preserve the successful
content write behavior and report the missing PDF rather than fabricating one.

Math authors must preserve the outline-to-ledger-to-lesson direction, teach one
installable idea at a time, and keep answer-key fields out of learner-visible
HTML. Biography authors must use a traceable public-domain source, paraphrase
rather than quote it, and preserve global section numbering across chapters.
Neither pipeline may substitute generic prose or unreviewed generated output for
its gate.

# known-limits

The legacy `sr-lesson` documentation is internally inconsistent with the active
math system. It refers to static serving and older exercise constraints while
the repository runtime is database-driven and the newer skill owns math work.
It remains available in the skill tree, so an agent that chooses it can produce
content misaligned with the current pipeline.

PDF generation is explicitly best effort in both active savers. A dependency or
browser-launch failure can persist a lesson or chapter without fresh PDF bytes.

The current application does not filter draft content from runtime queries, so
the skills' draft default does not itself prevent learner visibility. There is
also no runtime spaced-repetition scheduler; math review is entirely preplanned
in the generated deck.

# notes-for-ai

Choose an active child skill before starting content work. Use math courseware
for a ledger, math lesson, math deck, or math persistence task; use biography
reading for a public-domain story workflow. Do not use the legacy `sr-lesson`
for new work without an explicit compatibility reason and a review of its
incompatible assumptions. Read the child contract, relevant human guide, and
schema before authoring or running a saver.

Keep the author, gate, and persistence phases separate. A successful structural
script check is not a quality judgment, and a persuasive narrative or lesson is
not permission to bypass the saver. For math changes, trace title/order through
the human outline, ledger entry, saver metadata check, persisted id, curriculum
activation, and browser rendering. For biography changes, trace source
provenance, chapter order, global section boundaries, persisted Markdown,
server rendering, catalog stage grouping, and question behavior.

Verify a real saved result. Check the stored content in the app, the catalog
activation or chapter grouping, PDF availability, and a quiz fetch before and
after an answer to prove secrets remain hidden until the correct server phase.
When changing an active saver or contract, run its deterministic validator and
then exercise the runtime reader against a controlled database; the generated
artifact and the app consumer form one contract.
