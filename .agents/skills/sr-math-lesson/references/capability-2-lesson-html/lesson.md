# Capability 2 — Author one lesson's card-tree content JSONB

> **JSONB-first:** the author output is no longer a self-contained HTML file — it is the neutral card-tree `content` JSONB (`{ cards: [ { id, num, name, anchor, rev, body[], read_check[] } ] }`) plus a `zh` prose overlay (`{ node_id → { t, src_rev } }`). One card per genre section anchor, in order; each card carries a learner-visible `num` (编号) and a required section display name `name` (中文名 — the section label shown to the learner, e.g. `讲解` / `例题` / `为什么学这个`; pick a concrete, lesson-appropriate name, not just the anchor). Card `body[]` prose nodes are node-id refs whose text goes in the overlay; formula (KaTeX) and SVG nodes are neutral and inline in `content`. Author each **substantial** card's `read_check[]` (≥1 item: 概念课 motivation/model/anatomy/boundary/connections; 方法课 motivation/explain/examples/connections — not `oral`, not the 练习课 orientation); each read-check is answerable from that card alone. **Choice-only policy (STEMROBIN-25, reversible):** every read-check is a single-answer `choice` item with exactly 4 options (A/B/C/D) — one correct, three distractors of the same surface form as the answer, each wrong for a nameable misconception (no "无法确定 / 以上都不对", no meta-sentence 误区标签 options); do NOT author `input` (fill-in-the-blank) read-checks. `check-content.mjs` enforces this via `scripts/question-policy.mjs` (`CHOICE_ONLY`); the `input` code path is retained and re-enabled by flipping that flag. The answer KEY (`key.correct_index`; `key.accept` when `input` is re-enabled) stays in the neutral base, never in the overlay. Validate with `node .agents/skills/sr-math-lesson/scripts/check-content.mjs --content <c.json> --overlay <o.json> --genre <genre> --id <id>`. The HTML is rendered later by `render-lesson.mjs` from this JSONB — do NOT hand-write HTML.

From the ledger entry (read from the DB), author ONE lesson's content. Read `references/common/lesson-contract.md` (JSONB-first + Lesson genres) first — it defines the section anchors per genre and the vocabulary contract. The genre section structure, vocabulary contract, boundary-case coverage, model-first teaching, and self-check below still apply — they now describe the prose you place into the card `body[]` overlay text rather than raw HTML.

**Execution:** an independent subagent authors, given: the ledger JSON (its own entry + all earlier entries, so it knows exactly which terms are speakable), the contract, and `assets/lesson-template.html`. The orchestrator runs gate-2 before cap3/cap4. For a group of lessons, one author owns each page and authors run in parallel; one separate reviewer may gate the complete group, with a verdict for each page.

## Steps

0. **Blueprint first (the pre-authoring gate).** Before writing any prose, produce the lesson **blueprint** JSON (`references/common/lesson-contract.md` → "## Blueprint"): the `domain` + `domain_rationale` (your explicit representation judgment — is this lesson's content spatial or symbolic?), the per-section skeleton with a **figure plan** (which figures each section needs and a `why` for each — no preset count; geometry usually several, algebra usually none), and a `deck_plan`. Run `node .agents/skills/sr-math-lesson/scripts/check-blueprint.mjs <bp.json> --ledger <stage-ledger> --id <id>` (it fails a figure with no `why` = 废图) **and** self-review for 缺图 (a spatial claim/example/boundary with no figure). **Do not proceed to prose until the blueprint passes.**
1. Read your ledger entry: `genre`, `core_idea`, `introduces`, `consumes`, `boundary_cases`. These are commitments, not suggestions.
2. Compose the sections for your genre (概念课: motivation/model/anatomy/boundary/connections/oral; 方法课: motivation/explain/examples/connections/oral; 练习课: motivation only) inside the template's `<article class="sr-lesson">`, using the template's classes (`.sr-example/.sr-step/.sr-eg/.sr-pitfall/.sr-note/ol.sr-oral/.sr-answer/figure.sr-fig`). Fill the header EXACTLY per the contract's Header format: `{{TITLE}}` = `<stage>.<order> 标题` (e.g. `2.6 去括号`), `{{EYEBROW}}` = `数学 · <stage theme> · <genre>`. Fill `{{CONCEPT}} {{CHIPS}}`; leave no `{{}}`.
3. **Speak only taught language.** Terms you may use freely: your `consumes` + earlier `introduces` + `assumed`. Terms you must TEACH (define via the model, `<span class="sr-term">`, 正例/反例): your `introduces`. Any other technical term is forbidden — if you need one, stop and report a ledger problem instead of using it.
4. **Teach every `boundary_cases` entry** where it belongs (boundary section for 概念课, pitfalls for 方法课).
5. **Figures per the blueprint, as specs.** Author each planned figure as a body node `{ kind:"svg", spec:{…} }` (see the contract's "## Figures" + `scripts/figure.mjs`), not hand-drawn SVG — the tool computes correct coordinates. Draw figures where the blueprint says the content is spatial (a geometry `model`/`anatomy`/`boundary` usually needs them, often one per parsing example and a side-by-side pair for a boundary contrast), and **do not add figures an algebra lesson doesn't need**. Every figure's labels must appear in the card's prose (figure-text consistency, checked). 方法课 `explain`: derive the move from the principle — the learner must be able to answer "为什么可以这么做".
6. Self-check against the first cause: after this text, can the learner name parts fast, do the move, and say why? Is any paragraph a lecture with no instance? Is anything defined once and never shown?

Write to a scratch path (`<scratch>/<id>.html`). Do not author exercises here (cap3). Report: sections present, terms introduced/consumed used, boundary cases covered.

Run `node .agents/skills/sr-math-lesson/scripts/check-ledger.mjs <stage-ledger> --vocab <scratch>/<id>.html --id <lesson-id>` before handing the page to gate-2. Do not run browser visual QA here: the single final cap4 verification owns rendered KaTeX, PDF, and responsive layout after the stored lesson is complete.
